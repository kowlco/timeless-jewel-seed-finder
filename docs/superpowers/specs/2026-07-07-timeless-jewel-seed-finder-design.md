# Timeless Jewel Seed Finder — Design

- **Дата:** 2026-07-07
- **Статус:** дизайн одобрен посекционно, ожидает финального ревью спеки
- **Проект:** `D:\projs\timeless` (greenfield)

## 1. Назначение

Инструмент для Path of Exile, который по набору **желаемых результатов трансформации** timeless-джевела находит, какие сочетания «сид + сокет» дают их максимально. Пользователь задаёт цели (ноутаблы / кистоуны / статы мелких нод) с весами, инструмент выдаёт ранжированный список комбинаций `(джевел, вариант, сид, сокет)` по суммарному score, с разбивкой по целям, ссылкой на trade и визуализацией результата на дереве.

### Ключевой дифференциатор

Существующие тулы (Vilsol `vilsol.github.io/timeless-jewels`, зеркало `tjcalc.cc`) требуют **заранее выбрать сокет**. Здесь поиск идёт **сразу по всем валидным сокетам** (и, опционально, по всем вариантам джевела) — это главная причина существования проекта.

## 2. Scope

### MVP
- Поиск по **всем сокетам** выбранного джевела; вариант — тоже ось поиска (можно «по всем вариантам»).
- Взвешенный скоринг целей трёх видов: ноутабл, кистоун, стат мелкой ноды.
- Ранжированная таблица результатов с разбивкой по целям.
- **Trade-ссылки** (клиентская сборка URL к официальному трейду).
- **Текстовый разбор сокета** («нода X → становится Y»).
- **Визуальный рендер дерева** для выбранного результата (адаптация рендера Vilsol).

### Non-goals (фаза 2 или вне проекта)
- ❌ Моделирование DPS/выживаемости билда — это работа Path of Building. Ищем сиды только по покрытию пассивок.
- ⬜ Привязка к билду / импорт PoB / фильтр «только достижимые мной сокеты» — фаза 2.
- ⬜ Мультиязычные стат-переводы — фаза 2 (MVP только английский).
- ⬜ Индекс / инвертированный индекс как рантайм-оптимизация on-demand-режима — не нужен при выбранном офлайн-предрасчёте.
- ❌ Любой бэкенд/сервер — всё в браузере (статичный сайт).

## 3. Стек и архитектурные решения

Все решения приняты в ходе брейншторма:

- **Формат:** статичный сайт, деплой на GitHub Pages. Весь тулчейн на **TypeScript**.
- **UI:** **Svelte** + Vite.
- **Провенанс данных:**
  - Дерево пассивок — официальный экспорт GGG `grindinggear/skilltree-export` (`data.json`), пиннится по версии патча.
  - Таблицы трансформации (`AlternatePassiveSkills`, `AlternatePassiveAdditions`, `AlternateTreeVersions`, `PassiveSkills`, `Stats`) + стат-переводы — **само-экстракция из GGPK** через `pathofexile-dat` (TS) с публичного patch-CDN GGG. Установленный клиент игры не нужен.
  - Принцип: не зависим ни от чьего CDN и baked-артефактов. Допустимо переиспользовать только офиц. данные GGG и сырые механические дампы; интерпретацию (алгоритм) держим свою.
- **Алгоритм трансформации:** собственный **порт на TS**, читающий сырые таблицы. Корректность — golden-тесты.
- **Рантайм-поиск:** **офлайн-предрасчёт** всех `(джевел, вариант, сокет, сид)` в build-шаге, данные **шардятся по джевелу** и грузятся по требованию. Поиск в браузере = скоринг готовых агрегатов (мгновенно). Транформация в браузере используется только для рендера **одного выбранного** результата.
- **Рендер дерева:** адаптация открытого (MIT) Svelte-компонента рендера Vilsol под наши данные. Это UI-код, ортогональный корректности поиска.

## 4. Компоненты и интерфейсы

Изолированные модули, каждый тестируется отдельно. Две фазы жизни: **build-time** и **runtime**.

### Build-time
- **`tools/extract`** — Node CLI. Тянет дерево (`skilltree-export@<версия>`) + гоняет `pathofexile-dat` по patch-CDN GGG → сырые `.dat`-таблицы + стат-переводы. Нормализует в компактные JSON-ассеты `data/<версия>/` + `manifest.json` (версия дерева, патч, дата, чек-суммы). Schema-валидирует выход; при отсутствии ожидаемых таблиц/колонок — abort.
  - Интерфейс: `npm run extract -- --patch 3.27 --tree 3.27.0`.
- **`tools/precompute`** — Node CLI. Прогоняет `core/transform` по всем `(джевел, вариант, сокет, сид)` и строит агрегаты `(вариант, сокет, сид) → {исход → счётчик}`. Шардит по джевелу, сжимает (gzip/brotli). Выход коммитится или генерится в CI.
  - Интерфейс: `npm run precompute -- --jewel all`.

### Runtime — ядро (чистый TS, без DOM)
- **`core/types`** — единый источник констант: 5 джевелов, их варианты, диапазоны сидов (из данных, не хардкод), типы исходов. Значения выводятся из данных, где возможно.
- **`core/transform`** — `transform(jewel, variant, seed, nodeId, tables) → NodeOutcome`. Порт seeded-RNG + логика таблиц. Ядро корректности, покрыто golden-тестами. Чистая детерминированная функция. Используется: (а) в `tools/precompute`, (б) для рендера одного выбранного результата.
- **`core/radius`** — `nodesInRadius(socketId) → nodeId[]`. Предрасчёт по координатам дерева. Чистый.
- **`core/search`** — `score(shard, targets) → SearchResult[]`. Скорит предрасчитанные агрегаты против взвешенных целей, ранжирует, топ-N. `SearchResult = {jewel, variant, seed, socketId, score, breakdown: {targetId, count}[]}`.
- **`core/trade`** — `buildTradeUrl(jewel, variant, seed, league) → string`. Собирает trade-JSON-запрос + урл-энкод. Учитывает quirk Elegant Hubris (отображаемый сид = внутренний ×20) и имя лиги. Чистый.

### Runtime — данные, воркер, UI
- **`data/loader`** — ленивая загрузка шарда предрасчёта выбранного джевела + ассетов версии.
- **`worker/score.worker`** (опционально) — скоринг больших массивов агрегатов вне主 потока, если понадобится для плавности UI. При офлайн-предрасчёте тяжёлого цикла нет, так что воркер минимален/необязателен.
- **UI (Svelte):**
  - `JewelSelect` — джевел + вариант(ы).
  - `TargetPicker` — автокомплит по исходам + веса, опц. галка «цель обязательна» (hard-фильтр).
  - `ResultsTable` — ранжированные результаты, сортировка, trade-ссылка, кнопка «показать».
  - `TreeView` — адаптированный рендер Vilsol: выбранный результат на дереве, круг радиуса, трансформированные ноды подсвечены.
  - `SocketBreakdown` — текстовый список исходов сокета (дополняет рендер).
  - `App` — состояние/оркестрация.

### Поток данных
```
build-time:  GGG CDN → tools/extract → JSON-ассеты (коммит)
                                     → tools/precompute (через core/transform) → шарды по джевелу (коммит)
runtime:     выбор джевела → загрузка шарда → цели+веса → core/search (скоринг агрегатов) → таблица
             → выбор строки → core/transform(1 результат) → TreeView + SocketBreakdown + core/trade → ссылка
```

## 5. Корректность алгоритма

Стержень «без шорткатов». Оракул истины двухуровневый:

1. **Bulk cross-validation** против доверенной референс-реализации (тул Vilsol / PoB): офлайн генерим **репрезентативную выборку** входов `(джевел, вариант, сид, сокет)`, прогоняем через референс, записываем ожидаемые исходы, требуем **побайтового совпадения** нашего порта. Референс в рантайм **не тащим** — используем только как «линейку» на этапе разработки/тестов.
2. **Spot-checks** против известных значений из игры/вики на горстке эталонов, включая пограничные случаи — чтобы не зависеть целиком от чужого тула.

**Покрытие golden-фикстур (критерий готовности): репрезентативная выборка** — все варианты каждого джевела × сиды по краям и в середине диапазона × несколько сокетов × известные edge-cases. Фикстуры коммитятся, гоняются в CI.

## 6. Семантика поиска и скоринга

- Цели трёх видов: **ноутабл** (по id), **кистоун** (по id), **стат мелкой ноды** (по stat-id). У каждой — вес.
- `score(вариант, сид, сокет) = Σ (вес_цели × число_совпадений)`. У статов число совпадений может быть >1; ноутабл/кистоун обычно 0/1, формула общая.
- Ранжирование по убыванию, топ-N с разбивкой.
- Опционально (дёшево): «цель обязательна» → hard-фильтр строк без цели.

## 7. UI-поток

```
Выбор джевела + вариант(ы)
  → добавление взвешенных целей (автокомплит)
  → «Искать»
  → таблица результатов (сортировка, разбивка)
  → клик по строке
    → TreeView (сид+сокет на дереве, радиус, трансформированные ноды)
    → SocketBreakdown (текст)
    → кнопка trade-ссылки
```

## 8. Обработка ошибок (рантайм)
- Пустые цели / ноль результатов → понятное сообщение.
- Ошибка загрузки шарда/ассетов → явный error-state.
- Не загрузился спрайт иконок с CDN GGG → рендер без иконок/плейсхолдер, дерево не ломаем.
- Ошибка воркера (если используется) → показать, не падать молча.

## 9. Тестирование
- **`core/transform`** — golden-тесты (репрезентативная выборка, см. §5), тест детерминизма.
- **`core/radius`** — известный сокет → известный состав нод из дерева.
- **`core/trade`** — snapshot URL по каждому джевелу, отдельно Elegant Hubris ×20.
- **`core/search`** — фейковые агрегаты → ожидаемый рейтинг и разбивка.
- **`tools/extract`** — schema-валидация; наличие ожидаемых таблиц/колонок.
- **`tools/precompute`** — согласованность с `core/transform` на выборке; проверка размера шардов (см. §11).

## 10. Деплой
- Vite-сборка → статика → GitHub Pages через Actions (настройка `base` пути).
- Данные (ассеты + шарды) закоммичены → сборка сайта герметична, сеть не нужна.
- Браузеры: современные evergreen.

## 11. Риски и decision-gates
- **Размер предрасчёта (главный риск).** Оценка — десятки МБ на весь датасет / единицы–низкие десятки МБ на джевел (сжато), но не подтверждено. **Первая задача реализации — спайк**: реально замерить сжатый размер шарда на одном джевеле. Если размер неподъёмный — задокументированный **фолбэк на on-demand-поиск в воркере** (§3, вариант из брейншторма). Ядро (`core/transform`, `core/search`) переиспользуется в обоих режимах, так что фолбэк дешёвый.
- **Совпадение RNG в порте.** Снимается golden-тестами (§5); риск локализован в `core/transform`.
- **`pathofexile-dat` реально выдаёт нужные таблицы.** Проверяется отдельным шагом в начале реализации `tools/extract` (schema-driven, ожидаемо да, но подтверждаем).
- **Trade-stat-id сида по каждому джевелу.** Найти корректные id + обработать лигу и ×20; snapshot-тесты фиксируют.
- **Актуализация под лиги.** Экстракция + предрасчёт — плановый batch раз в лигу.

## 12. Фазирование
- **Фаза 1 (MVP):** §11-спайк размера → `tools/extract` → `core/transform` + golden → `tools/precompute` → `data/loader` + `core/search` → UI (JewelSelect, TargetPicker, ResultsTable) → `core/trade` → SocketBreakdown → `TreeView` (адаптация Vilsol) → деплой на Pages.
- **Фаза 2:** привязка к билду / импорт PoB и фильтр достижимых сокетов; мультиязычность; доп. UX по вкусу.
